课题：AgentHub - 多 Agent 协作平台

---
课题背景
通过对话式交互创建网页、Workflow 等产物。本课题要求学生构建一个该业务的简化实战版：多 Agent 协作平台（AgentHub）。
平台采用 IM 聊天作为核心交互范式。用户像使用飞书/微信一样，通过新建对话、发送消息的方式与不同 AI Agent 进行交互。每个 Agent 就是一个"聊天对象"，用户可以：
- 新建对话：创建一个新的聊天会话，选择或指定要对话的 Agent（如 Claude Code、Codex、OpenCode 等）
- 多会话并行：同时开启多个对话窗口，分别与不同 Agent 交流不同任务（类似 IM 的多个聊天窗口）
- 群聊协作：在一个对话中 @ 多个 Agent，由主 Agent（Orchestrator）自动协调分工，多个 Agent 像群聊成员一样依次回复各自的产出
- 上下文连续：每个对话保持完整的聊天历史，Agent 能基于历史消息理解上下文，支持多轮迭代修改
- 产物内联：Agent 的回复不仅是文字，还可以内联展示代码 Diff、网页预览卡片、文件附件等富媒体产物，用户可直接在聊天流中预览和操作
平台同时接入市面主流 Agent 平台（Claude Code、Codex、OpenCode 等），通过统一的适配器层屏蔽 API 差异，并支持用户自建 Agent。所有 Agent 产出（代码、网页、文档、PPT 等）支持实时预览、代码二次编辑和一键部署发布。

一、我建议的最终产品定位

产品名：AgentHub

一句话定位：

一个 IM 形态的多 Agent 协作平台，用户像拉群聊一样组织 Claude Code、Codex、OpenCode 等 Agent 协作生成网页、代码和 Workflow，并在聊天流中实时查看 Diff、预览和部署状态。

核心不是“又一个 Dify”，也不是“又一个 Claude Code GUI”。

更准确的差异点是：

AgentHub = Agent 群聊工作台 + Coding Agent 适配器 + 产物预览面板。

5. Electron 还是 Tauri？

我的判断：

课程项目 + 高性能桌面壳：优先 Tauri。
如果你们要极限省事、Node 集成最简单：Electron 也可以，但我不推荐作为最终形态。

Electron 的优势

Electron 官方定位就是用 Chromium + Node.js 让一套 JavaScript 代码构建 Windows、macOS、Linux 桌面应用。它的生态成熟，Node 集成直接，打包开发者工具类产品很顺。
Electron
+1

适合：

快速打包；

直接用 Node 后端；

需要大量 Node native 能力；

团队不想碰 Rust；

只做 Windows/macOS/Linux 桌面端。

缺点也明显：

包体大；

内存占用高；

每个 App 自带 Chromium；

移动端基本不覆盖；

对“高性能轻量桌面壳”的叙事不如 Tauri。

Tauri 的优势

Tauri 2 官方定位是小型、快速、安全的跨平台应用框架，支持 Linux、macOS、Windows、Android、iOS，并且可以复用任意前端框架。
Tauri
+1
 它还支持 sidecar 外部二进制，官方文档给出了通过 @tauri-apps/plugin-shell 启动 sidecar 的方式，这对你把 agenthubd 或 runner 包进桌面端很有价值。
Tauri
+1

适合：

桌面端轻量；

本地文件访问；

管理本地进程；

包装本地 daemon；

安全权限控制；

做“高性能、现代、轻量”的产品叙事。

缺点：

要碰 Rust / Tauri 配置；

Node 后端不能像 Electron 那样天然塞进去；

mobile 虽然 Tauri 2 支持，但调试和生态成熟度不如 Web/PWA/Capacitor；

如果你把 Node daemon 作为 sidecar 打包，构建链会复杂一点。

我的最终选择
桌面端：Tauri
Web 端：Next.js / Vite React
移动端：PWA 优先，Capacitor 次选
核心执行层：agenthubd，本地/远程都能跑

不要试图用 Tauri 一套直接通吃所有执行逻辑。Tauri 只做壳和本地权限桥，真正的 Agent 执行放在 agenthubd。

6. 移动端怎么做？Android / iOS 能不能跑 Claude Code？

移动端只做控制台，不做执行端。

也就是说：

Android / iOS App
      ↓
查看会话
发送指令
审批工具调用
看 Diff
看预览
停止任务
      ↓
连接本地/远程 Runner

不要让 Android / iOS 本机跑 Claude Code。现实问题很多：

iOS 沙箱限制强；

CLI 工具链不完整；

文件系统权限弱；

长时间后台进程不稳定；

App Store 审核风险；

移动设备不适合跑 pnpm install、构建、测试、Docker。

移动端正确路线是：

PWA / Capacitor App → AgentHub Server / Local Runner / SSH Runner

Capacitor 官方定位就是把 Web App 变成可在 iOS、Android 等平台原生运行的应用，并尽量保持 Web 技术栈，同时访问原生能力。
Capacitor
+1
 所以如果你们后期要 Android/iOS 包，Capacitor 比 Tauri Mobile 更适合课程项目。

11. 最终推荐方案

你的 AgentHub 网络架构应该这样写进技术文档：

AgentHub 采用控制面与执行面分离的网络架构。

控制面由 AgentHub Core 提供，负责 Web UI 接入、用户会话、消息同步、Agent 编排和权限控制。客户端通过 HTTPS 与 WebSocket 连接 Core，以获得 IM 式实时交互体验。

执行面由 AgentHub Runner 提供，负责连接 Claude Code、Codex、OpenCode 等本地或远程 Agent 进程，执行代码生成、文件修改、测试运行和网页预览。Runner 不直接暴露公网，优先采用本地回环、SSH Tunnel 或主动反连方式与 Core 通信。

平台支持三种运行模式：
1. 本地模式：Core 与 Runner 运行在用户本机，适合个人开发和课程 Demo。
2. SSH 远程模式：Core 通过 SSH Tunnel 连接远程 Runner，适合服务器执行和高性能任务。

最合适的网络架构是：

本地优先，控制面和执行面分离；UI 通过 HTTPS/WebSocket 连接 Core，Core 通过 local / SSH tunnel / reverse WebSocket 管理 Runner；Runner 负责真实执行 Claude Code/Codex/OpenCode，并且默认不暴露公网。

4. 公网 Server 会不会延迟高？

分情况。

4.1 聊天控制流：延迟基本不是问题

用户发一句话、Agent 回一条状态、任务 started/done，这些都是小包。

即使公网 Server RTT 是 80ms、150ms，对体验影响也不大。

UI → Cloud Server → Runner

对普通消息来说没问题。

4.2 日志流：有影响，但可以靠批处理解决

Claude Code / Codex 会疯狂输出日志。如果每一行都走公网 WebSocket，前端会卡，Server 也会有压力。

正确做法：

Runner stdout
  ↓
50ms / 100ms buffer
  ↓
run.log.batch
  ↓
Server
  ↓
UI

不要这样：

每一行日志发一个 WebSocket event

推荐事件：

TypeScript
{
  type: "run.log.batch",
  runId: "run_123",
  lines: [
    "reading file...",
    "editing component...",
    "running test..."
  ]
}

这样公网也能接受。

4.3 文件 / Diff / 预览：公网 Server 可能明显变慢

真正容易慢的是这些：

文件树
图片

构建产物

2. Memory 分四层

你不要只做一个笼统的 “memory”。AgentHub 里至少要分四类。

  每个 Agent 自己的身份、习惯、能力、长期经验

## Architecture
- UI 连接 AgentHub Server。

这个设计很好讲，也方便 AI 协作开发记录。

4. Markdown 怎么用最合适

Markdown 很适合做 可读、可编辑、可版本管理的 Memory。
但聊天消息不要全部导出成 Markdown 当主存储。

推荐：

消息历史：SQLite 主存储
Memory 文档：Markdown 主存储 + SQLite 索引
运行日志：JSONL / log 文件主存储 + SQLite 索引
Artifact：文件主存储 + SQLite 索引

也就是：

数据	主存储
Message	SQLite
Conversation	SQLite
Agent Profile	SQLite + Markdown
Project Memory	Markdown + SQLite 索引
Agent Memory	Markdown + SQLite 索引
Logs	文件
Diff	patch 文件
Preview	URL / 文件
Artifact 元数据	SQLite

不要把数据库和 Markdown 二选一。
它们用途不同：

SQLite	Markdown
查询快	人能读
适合 IM 消息	适合项目规范
适合状态管理	适合长期知识
适合索引	适合 Git 管理
机器友好	AI 友好、人友好
5. 项目级 Memory 如何自动更新

每次 Agent 完成一个重要任务后，Server 可以触发 Memory Update。

流程：

Agent Run 完成
 ↓
Server 收集：
- 用户需求
- Agent 输出
- 修改文件
- Diff
- Reviewer 结论
 ↓
Memory Writer 生成更新建议
 ↓
用户确认
 ↓
写入 .agenthub/decisions.md 或 project.md
 ↓
重新索引 Memory

不要让 Agent 每次自动乱写 Memory，容易污染。

更稳的做法是生成“记忆更新建议卡片”：

[Memory Update Suggestion]

建议写入 project.md：

- AgentHub Server 负责会话、消息、调度和 Memory。
- Runner 只负责执行 Agent CLI，不保存长期记忆。
- 本地模式下 Server 和 Runner 可同进程部署。

[接受] [编辑后接受] [拒绝]

这样可控。

Conversation	IM 会话	“AgentHub 架构群”
Thread	会话中的子讨论	针对某条消息展开讨论
AgentSession	某个 Agent 在某会话里的运行会话	Claude Code 在这个项目里的 session
Run	一次具体执行	Claude Code 这次生成登录页
Message	聊天消息	用户/Agent 的发言
Artifact	产物	Diff、文件、预览、部署结果

关系：

Project
 └── Conversation
      ├── Message
      ├── Thread
      ├── AgentSession
      └── Run
           └── Artifact

14. 最终设计结论

你们可以这样定：

AgentHub 的持久化由 Server 统一负责，Runner 不保存长期状态。

系统中的 Memory 分为 Global Memory、Project Memory、Agent Memory 和 Conversation Memory 四层。Project Memory 与 Agent Memory 使用 Markdown 保存，便于人类阅读、AI 引用和 Git 版本管理；消息、会话、线程、联系人、运行记录和 Artifact 元数据使用 SQLite 持久化。

IM 系统以 Conversation 为核心，单聊和群聊都是 Conversation。群聊成员通过 ConversationMember 管理，成员可以是用户或 Agent。@Agent 通过 mentions 字段解析，由 Server 调度对应 Agent。线程 Thread 用于对某条消息展开子讨论，避免主群聊被细节刷屏。

Agent 是持久化对象，包含 Profile、System Prompt、Tools、Permissions、Memory 和 Runner Binding。每个 Agent 可以拥有自己的长期 Markdown Memory，也可以在不同 Conversation 中保持独立 AgentSession。

每次调用 Agent 前，Server 的 Context Builder 会根据当前项目、会话、线程、Agent、pin 消息、最近消息、会话摘要和相关 Artifact 构造上下文，再交给 Runner 执行。

最短的架构链路就是：

Conversation / Group / Thread
        ↓
Message Persistence
        ↓
Context Builder
        ↓
Agent Memory + Project Memory
        ↓
Runner Execution
        ↓
Artifact + Memory Update

这套设计既像 IM，又能支撑多 Agent、项目级记忆、Markdown 协作、线程、上下文连续和 Agent 持久化。
