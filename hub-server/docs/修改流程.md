# hub-server 性能优化实施流程

> 本文件按**阶段**指导实施，每个阶段是一次可独立提交、可独立验证、可独立回滚的最小变更单元。
>
> **使用方式**：按阶段顺序执行；每个阶段「不通过验收 → 不进入下一阶段」。
> **配套文档**：详细的代码改动规格见同目录 `修改需求.md`（以下简称「需求」）。
> **改动编号**：R1–R10 与需求文档一致。

---

## 阶段总览

| 阶段 | 包含改动 | 关键目标 | 风险等级 |
|------|----------|----------|----------|
| 阶段 0 | 准备工作 | 基线测试、备份、依赖准备 | — |
| 阶段 1 | R3 + R4 + R1 + R2 | 可观测性 + 基础硬化 | 低 |
| 阶段 2 | R5 | 异步日志 | 中 |
| 阶段 3 | R6 | EventBus worker pool | 中 |
| 阶段 4 | R8 + R9 | 业务数据缓存 + singleflight | 中 |
| 阶段 5 | R7 | Seq 迁移到 Redis | **高** |
| 阶段 6 | R10 | 转发并发化 | 低 |

**步骤 7：旧会话 seq 同步**

在 `cmd/server-hub/main.go` 中 `cache.InitRedis` 成功之后插入：

```go

```

并实现：

```go
func syncLegacySeq() {
    ctx := context.Background()
    var sessions []model.Session
    if err := repository.DB.Select("id, next_seq").Where("next_seq > 0").Find(&sessions).Error; err != nil {
        slog.Warn("legacy seq sync skipped", "error", err)
        return
    }
    for _, s := range sessions {
        _ = cache.InitSeqIfAbsent(ctx, s.ID, s.NextSeq)
    }
    slog.Info("legacy seq synced", "count", len(sessions))
}
```

> 该函数使用 `SetNX`，对已有 key 不会覆盖，幂等可重入。

**步骤 8：保留旧 `AllocateSeqID` 函数**

`internal/repository/message.go:53-60` 的 `AllocateSeqID` **不要删除**：留作 fallback 与历史代码引用。但 service 层不再调用。

### 5.3 灰度建议

如生产环境敏感：

1. 在 service 层加 feature flag（环境变量 `HUB_SEQ_BACKEND=redis|db`），默认 `db`。
2. 测试环境跑 `redis` 一周，观察 `messages` 表无重复 `(session_id, seq_id)` 唯一约束冲突。
3. 灰度 10% → 50% → 100%。
4. 全量后下一个迭代删除 flag 与旧路径。

> **本期不强制做 feature flag**。决策权在用户。

### 5.4 验收

| 项 | 方法 | 预期 |
|----|------|------|
| 编译通过 | `go build ./...` | 无错误 |
| 单会话强一致 | 1k goroutine 同时发消息到同一 session | 所有 seq 连续递增、无重复、无空洞 |
| 唯一约束兜底 | 人为 `redis-cli SET session:seq:xxx 0` 倒退 | DB 唯一索引冲突，service 返回错误而非脏数据 |
| Redis 故障 | 杀掉 Redis | 发消息 5xx，恢复后正常（service 不应崩溃） |
| Redis 重启不丢数据 | 重启 Redis 容器 | seq 不倒退（验证 AOF 生效） |
| 单会话 P99 延迟 | 对比基线 | 下降 ≥ 70% |
| 历史会话 | 老 session 收发消息 | seq 从 DB 中的 `next_seq` 接续，不归零 |

### 5.5 回滚

如生产已切流：

1. 回滚代码：`git revert <stage5-commit>`，重新走 DB 行锁路径。
2. 把 Redis 中的 seq 写回 DB（保险起见）：

```bash
redis-cli --scan --pattern 'session:seq:*' | while read k; do
    sid="${k#session:seq:}"
    seq=$(redis-cli GET "$k")
    psql -c "UPDATE sessions SET next_seq = $seq WHERE id = '$sid' AND next_seq < $seq"
done
```

3. 重启服务。

## 实施纪律

1. **每个阶段独立 commit、独立验收、独立可回滚**。不要把多个阶段揉到一个 PR。
2. **不在性能优化分支里做无关重构**。看到需要清理的代码，单独开 issue。

4. **基线数据必须在阶段 0 采集**。事后补不上。
