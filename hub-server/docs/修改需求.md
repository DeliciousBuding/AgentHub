## R1. HTTP Server 超时配置

### 现状
- 文件：`cmd/server-hub/main.go:251`
- 当前实现：`r.Run(fmt.Sprintf(":%d", cfg.Server.Port))`
- 等价于 `http.ListenAndServe`，所有超时为零值（无限）。

### 问题
慢客户端（不发完 header、不读响应）会无限占用 goroutine；TCP 连接泄漏后无 IdleTimeout 回收。

### 改动方案
将 `r.Run(...)` 替换为显式 `http.Server` 并配置以下参数：

```go
srv := &http.Server{
    Addr:              fmt.Sprintf(":%d", cfg.Server.Port),
    Handler:           r,
    ReadHeaderTimeout: 5 * time.Second,
    ReadTimeout:       30 * time.Second,
    WriteTimeout:      60 * time.Second,
    IdleTimeout:       120 * time.Second,
    MaxHeaderBytes:    1 << 20, // 1 MiB
}

go func() {
    if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
        slog.Error("server failed", "error", err)
        os.Exit(1)
    }
}()

// 优雅停机
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
<-quit
ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
defer cancel()
_ = srv.Shutdown(ctx)
```

注意：
- `WriteTimeout` 设 60s，避免**附件下载大文件**被截断。如有更长下载需求，需为下载路由单独裸 hijack 或拆分到独立子路由（本期不做）。
- WebSocket 由 `coder/websocket` 自行处理 Upgrade 后的 IO 超时，不受 `ReadTimeout` 影响（升级后接管底层连接）。

### 涉及新增 import
- `net/http`、`os/signal`、`syscall`、`context`、`time`

### 验收标准
- 启动后 `curl --max-time 35 http://localhost:8080/...` 慢请求被服务端断开。
- `kill -SIGTERM` 进程能在 10s 内退出且不丢正在处理的请求。

## R2. Redis 连接池扩容

### 现状
- 文件：`internal/cache/redis.go:15-21`
- 当前实现：`PoolSize: 10`，无 MinIdleConns、无超时、无重试。

### 问题
WebSocket 上下线触发 `cache.SetRoute` / `cache.DeleteRoute`，加上后续要引入的业务缓存读写、seq 分配等，10 个连接极易排队。

### 改动方案
更新 `redis.NewClient` 配置：

```go
rdb := redis.NewClient(&redis.Options{
    Addr:            cfg.Addr(),
    Password:        cfg.Password,
    DB:              cfg.DB,
    PoolSize:        cfg.PoolSize,        // 新增配置项，默认 100
    MinIdleConns:    cfg.MinIdleConns,    // 新增，默认 10
    MaxRetries:      3,
    DialTimeout:     2 * time.Second,
    ReadTimeout:     1 * time.Second,
    WriteTimeout:    1 * time.Second,
    PoolTimeout:     4 * time.Second,
    ConnMaxIdleTime: 10 * time.Minute,
})
```

### 配套配置改动
1. `internal/config/config.go` 的 `RedisConfig` 增加字段：
   ```go
   PoolSize     int `mapstructure:"pool_size"`
   MinIdleConns int `mapstructure:"min_idle_conns"`
   ```
2. `configs/config.yaml` 与 `configs/config.docker.yaml` 增加：
   ```yaml
   redis:
     ...
     pool_size: 100
     min_idle_conns: 10
   ```
3. `InitRedis` 内部对零值兜底：`if cfg.PoolSize == 0 { cfg.PoolSize = 100 }`，同理 `MinIdleConns`。

### 验收标准
- 启动日志打印 `pool_size` 与 `min_idle_conns`。
- 压测 1k QPS 下 `redis-cli CLIENT LIST` 看到的连接数 ≥ 10。

## R5. 异步结构化日志（zap）

### 现状
- 文件：`internal/log/log.go:23`
- 当前实现：`slog.NewJSONHandler(os.Stdout, ...)`，每条日志同步写 stdout，并发下 stdout 锁竞争严重。
- Gin 自带 Logger 也同步写 stdout（在 R4 中已被移除）。

### 问题
高并发下日志同步写是显著瓶颈；同时 slog 默认 handler 在 hot path 上有 reflect 开销。

### 改动方案

**5.1 引入 zap + lumberjack**

```
go.uber.org/zap v1.27.0
gopkg.in/natefinch/lumberjack.v2 v2.2.1
```

**5.2 重写 `internal/log/log.go`**

保留 `Init(cfg *config.ServerConfig)` 签名不变，输出对接到 `slog.Default()`（让其他文件无需改动）。两种实现方案二选一：

- **方案 A（推荐）**：直接将 `slog` 切换为基于 zap 的 handler（使用 `go.uber.org/zap/exp/zapslog`）。
- **方案 B**：直接把项目中全部 `slog.Xxx` 调用替换为 `zap.L().Xxx`。改动面大，本期不选。

采用方案 A 的关键代码：

```go
package log

import (
    "io"
    "os"

    "go.uber.org/zap"
    "go.uber.org/zap/exp/zapslog"
    "go.uber.org/zap/zapcore"
    "gopkg.in/natefinch/lumberjack.v2"
    "log/slog"

    "github.com/agenthub/server-hub/internal/config"
)

var L *zap.Logger

func Init(cfg *config.ServerConfig) {
    level := parseLevel(cfg.LogLevel)

    var ws zapcore.WriteSyncer
    if cfg.LogFile != "" {
        ws = zapcore.AddSync(&lumberjack.Logger{
            Filename:   cfg.LogFile,
            MaxSize:    100,  // MiB
            MaxBackups: 10,
            MaxAge:     14,   // days
            Compress:   true,
        })
    } else {
        ws = zapcore.AddSync(os.Stdout)
    }

    encCfg := zap.NewProductionEncoderConfig()
    encCfg.TimeKey = "ts"
    encCfg.EncodeTime = zapcore.ISO8601TimeEncoder
    enc := zapcore.NewJSONEncoder(encCfg)

    core := zapcore.NewCore(enc, ws, level)
    L = zap.New(core, zap.AddCaller(), zap.AddStacktrace(zapcore.ErrorLevel))

    slog.SetDefault(slog.New(zapslog.NewHandler(L.Core())))
}

func Sync() { _ = L.Sync() }

func parseLevel(s string) zapcore.Level {
    switch s {
    case "debug": return zapcore.DebugLevel
    case "warn":  return zapcore.WarnLevel
    case "error": return zapcore.ErrorLevel
    default:      return zapcore.InfoLevel
    }
}

var _ io.Writer = (*lumberjack.Logger)(nil)
```

`main()` 末尾 defer `log.Sync()`。

## R6. EventBus worker pool

### 现状
- 文件：`internal/service/eventbus.go:30-43`
- 每次 `Publish` 为每个订阅者新建一个 goroutine。

### 问题
高频事件（如大群消息广播 `message.new`）瞬间产生数百 goroutine，调度与 GC 抖动。

### 改动方案

**6.1 引入 worker pool**

```
github.com/panjf2000/ants/v2 v2.10.0
```

**6.2 重写 `Bus`**

```go
package service

import (
    "context"
    "sync"

    "github.com/panjf2000/ants/v2"
    "github.com/agenthub/server-hub/internal/metrics"
)

type Bus struct {
    mu       sync.RWMutex
    handlers map[string][]EventHandler
    pool     *ants.Pool
}

func NewBus() *Bus {
    pool, err := ants.NewPool(1024,
        ants.WithNonblocking(false),
        ants.WithPanicHandler(func(p interface{}) {
            slog.Error("eventbus panic", "panic", p)
        }),
    )
    if err != nil {
        panic(err)
    }
    return &Bus{
        handlers: make(map[string][]EventHandler),
        pool:     pool,
    }
}

func (b *Bus) Publish(ctx context.Context, event Event) {
    b.mu.RLock()
    handlers := append([]EventHandler{}, b.handlers[event.Type]...)
    handlers = append(handlers, b.handlers["*"]...)
    b.mu.RUnlock()

    metrics.EventBusQueueLen.WithLabelValues(event.Type).Set(float64(b.pool.Running()))

    for _, h := range handlers {
        h := h
        _ = b.pool.Submit(func() {
            defer func() { recover() }()
            h(ctx, event)
        })
    }
}

func (b *Bus) Close() { b.pool.Release() }
```

`main()` 末尾 defer `bus.Close()`。

### 关键决策
- 池大小 1024：等于「同时处理 1024 个事件分发任务」上限。WS 推送本身有 send channel buffer，溢出已有 drop 逻辑，无需在此层兜底。
- `WithNonblocking(false)`：池满时 `Submit` 阻塞调用方，给 publish 端反压。如果发现 publish 路径被阻塞过久（监控指标），改为 `true` + 落地到 Redis 离线队列。**本期保持 false**。

### 验收标准
- `runtime.NumGoroutine()` 在压测下保持稳定（不再随事件数线性增长）。
- `eventbus_queue_len{type="message.new"}` 指标可见。

## R7. Seq 生成迁移到 Redis

### 现状
- 文件：`internal/repository/message.go:53-60`：`AllocateSeqID` 通过 `UPDATE sessions SET next_seq = next_seq + 1 ... RETURNING next_seq` 在 PostgreSQL 行锁上分配。
- 调用点：`internal/service/message.go:135`（SendMessage）、`internal/service/message.go:446`（ForwardMessage）。

### 问题
**这是当前最严重的单会话 QPS 瓶颈**。同一会话所有发消息请求都串行化在 `sessions` 表的同一行上。

### 改动方案（采用 Redis INCR 方案）

**7.1 新增 `internal/cache/seq.go`**

```go
package cache

import (
    "context"
    "fmt"
    "strconv"
)

const seqKeyPrefix = "session:seq:"

func seqKey(sessionID string) string { return seqKeyPrefix + sessionID }

func AllocateSeq(ctx context.Context, sessionID string) (int64, error) {
    return RDB.Incr(ctx, seqKey(sessionID)).Result()
}

func InitSeqIfAbsent(ctx context.Context, sessionID string, seq int64) error {
    _, err := RDB.SetNX(ctx, seqKey(sessionID), seq, 0).Result()
    return err
}

func PeekSeq(ctx context.Context, sessionID string) (int64, error) {
    s, err := RDB.Get(ctx, seqKey(sessionID)).Result()
    if err != nil {
        return 0, err
    }
    return strconv.ParseInt(s, 10, 64)
}

func _ignore() string { return fmt.Sprint("") }
```

**7.2 改造 SendMessage / ForwardMessage**

发消息事务中不再调用 `repository.AllocateSeqID`。新流程：

```go
seq, err := cache.AllocateSeq(ctx, sessionID)
if err != nil {
    return nil, err
}
msg.SeqID = seq

err = s.db.Transaction(func(tx *gorm.DB) error {
    if err := repository.InsertMessage(tx, msg); err != nil {
        return err
    }
    return repository.TouchSessionLastMessage(tx, sessionID)
})
```

需要新增 `repository.TouchSessionLastMessage(tx, sessionID)`：
```go
func TouchSessionLastMessage(db *gorm.DB, sessionID string) error {
    return db.Model(&model.Session{}).Where("id = ?", sessionID).
        Update("last_message_at", gorm.Expr("NOW()")).Error
}
```

**7.3 冷启动 / 缺 key 兜底**

Redis key 缺失时，从 DB `sessions.next_seq` 读取并 SetNX 初始化。这一逻辑放在 `AllocateSeq` 调用点：

```go
func (s *MessageService) allocateSeq(ctx context.Context, sessionID string) (int64, error) {
    seq, err := cache.AllocateSeq(ctx, sessionID)
    if err == nil {
        return seq, nil
    }
    if !errors.Is(err, redis.Nil) {
        return 0, err
    }
    sess, err := repository.GetSessionByID(s.db, sessionID)
    if err != nil {
        return 0, err
    }
    _ = cache.InitSeqIfAbsent(ctx, sessionID, sess.NextSeq)
    return cache.AllocateSeq(ctx, sessionID)
}
```

**注意**：`go-redis` 的 `INCR` 在 key 不存在时**不会返回 `redis.Nil`**，而是从 0 开始递增并返回 1。因此**实际兜底逻辑必须在创建 session 时同步 SetNX**。修正方案：

- 在 `service.SessionService.CreatePrivate` / `CreateGroup`（创建 session 的入口）插入 session 行后，**立即调用** `cache.InitSeqIfAbsent(ctx, sessionID, 0)`。
- 旧数据兼容：在 `InitDB` 之后，运行一次性扫描：把 `sessions` 表中 `next_seq > 0` 的会话 SetNX 到 Redis（key 不存在时才设置），避免旧会话从 0 开始重复 seq。

**7.4 消息表唯一约束兜底**

数据库层应已有 `UNIQUE(session_id, seq_id)`（请通过 `migrations/` 确认；如无，新增迁移）。即使 Redis 偶发脏数据，也能通过唯一索引报错回滚。

**7.5 持久化**

Redis 必须开启 AOF（appendonly yes）或配合定期回写到 PostgreSQL `sessions.next_seq`：

- AOF 方案：直接修改 Redis 配置（`deployments/` 与生产环境）。
- 回写方案：新增后台 goroutine 每 30s 把 Redis 中的 seq Peek 出来 UPDATE 到 DB。**本期采用 AOF 方案**，免去额外回写代码。

### 涉及文件清单
- 新增 `internal/cache/seq.go`

## R8. 业务数据缓存层

### 现状
所有业务读请求直接打 PostgreSQL，无任何业务缓存。

### 问题
读密集型接口（会话列表、群成员、用户信息）DB CPU 与连接占用高。

### 改动方案

**8.1 新增 `internal/cache/data.go`，提供通用「JSON 缓存 + singleflight」工具**

```go
package cache

import (
    "context"
    "encoding/json"
    "errors"
    "math/rand"
    "time"

    "github.com/redis/go-redis/v9"
    "golang.org/x/sync/singleflight"
)

var sf singleflight.Group

func GetOrLoad[T any](ctx context.Context, key string, ttl time.Duration, loader func() (T, error)) (T, error) {
    var zero T

    if s, err := RDB.Get(ctx, key).Result(); err == nil {
        var v T
        if json.Unmarshal([]byte(s), &v) == nil {
            return v, nil
        }
    } else if !errors.Is(err, redis.Nil) {
        // Redis 故障时降级为直接查源
        return loader()
    }

    v, err, _ := sf.Do(key, func() (any, error) {
        loaded, err := loader()
        if err != nil {
            return zero, err
        }
        b, _ := json.Marshal(loaded)
        // 防雪崩：TTL ± 10% 抖动
        jitter := time.Duration(rand.Int63n(int64(ttl) / 10))
        _ = RDB.Set(ctx, key, b, ttl+jitter).Err()
        return loaded, nil
    })
    if err != nil {
        return zero, err
    }
    return v.(T), nil
}

func Invalidate(ctx context.Context, keys ...string) {
    if len(keys) == 0 { return }
    _ = RDB.Del(ctx, keys...).Err()
}
```

**注意**：Go 1.18+ 支持泛型，`go.mod` 已是 1.25.6 满足。

| 会话元数据 | `session:meta:{sessionID}` | 10 分钟 | `UpdateGroupInfo`、`Dissolve`、`TransferOwner` |
| 会话活跃成员列表 | `session:members:{sessionID}` | 5 分钟 | `AddMembers`、`RemoveMember`、`Leave`、`Dissolve` |

## R10. 消息转发并发化

### 现状
- 文件：`internal/service/message.go:392-461`（`ForwardMessage`）
- 当前实现：对 `targetSessionIDs` 串行执行「校验 → 事务 → 发布事件」。

### 问题
转发到 N 个会话耗时 ≈ N × 单会话耗时，转发到群多时延迟极高。

### 改动方案

使用 `golang.org/x/sync/errgroup` 并发执行，限制并发度：

```go
func (s *MessageService) ForwardMessage(ctx context.Context, userID, msgID string, targetSessionIDs []string) error {
    msg, err := repository.GetMessageByID(s.db, msgID)
    if err != nil {
        return errcode.MsgNotFound
    }
    if active, err := repository.IsMemberActive(s.db, msg.SessionID, model.MemberTypeUser, userID); err != nil || !active {
        return errcode.SessionNotMember
    }

    g, gctx := errgroup.WithContext(ctx)
    g.SetLimit(8)

    for _, sid := range targetSessionIDs {
        sid := sid
        g.Go(func() error {
            return s.forwardOne(gctx, userID, msg, sid)
        })
    }
    return g.Wait()
}

func (s *MessageService) forwardOne(ctx context.Context, userID string, src *model.Message, sessionID string) error {
    active, err := repository.IsMemberActive(s.db, sessionID, model.MemberTypeUser, userID)
    if err != nil {
        return err
    }
    if !active {
        return errcode.SessionNotMember
    }

    sess, err := repository.GetSessionByID(s.db, sessionID)
    if err != nil {
        return errcode.SessionNotFound
    }
    if sess.Dissolved {
        return errcode.SessionDissolved
    }

    if sess.Type == model.SessionTypePrivate {
        other, err := repository.GetOtherMemberInPrivate(s.db, sessionID, userID)
        if err != nil { return err }
        if other != nil {
            blocked, err := repository.IsBlockedBy(s.db, other.MemberID, userID)
            if err != nil { return err }
            if blocked { return errcode.MsgBlockedByReceiver }
        }
    }

    seq, err := s.allocateSeq(ctx, sessionID) // 复用 R7 的 helper
    if err != nil { return err }

    forwarded := &model.Message{
        SessionID:   sessionID,
        ClientMsgID: uuidv7.Must(),
        SenderType:  model.SenderTypeUser,
        SenderID:    userID,
        ContentType: src.ContentType,
        Content:     src.Content,
        SeqID:       seq,
    }

    if err := s.db.Transaction(func(tx *gorm.DB) error {
        if err := repository.InsertMessage(tx, forwarded); err != nil { return err }
        return repository.TouchSessionLastMessage(tx, sessionID)
    }); err != nil {
        return fmt.Errorf("forward to session %s: %w", sessionID, err)
    }

    s.bus.Publish(ctx, Event{Type: "message.new", Payload: forwarded})
    return nil
}
```

### 关键决策
- `g.SetLimit(8)`：避免一次性占用全部 DB 连接。
- 任一会话失败立刻取消其他（errgroup 默认行为）。如希望失败不影响其他会话，把 `g.Go` 内部错误改为只 log 不返回。**本期采用默认行为（任一失败即终止）**。
- 依赖 R7 完成后才能用 `s.allocateSeq`。

### 验收标准
- 转发到 5 个会话耗时 ≤ 1.5 × 单会话耗时。
